# 웹클론 v2 기획 명세서 — 사이트 구조 클론 · AI 리브랜딩 · WYSIWYG 발행

작성일: 2026-07-09 · 대상: `shinkang888-code/webclone`
상태: 기획(구현 착수 전) · 문서 버전: v2.0

---

## 1. 무엇을 바꾸는가 (제품 재정의)

| | 현재 (v1) | 목표 (v2) |
|---|---|---|
| 캡처 대상 | 페이지 이미지·HTML 스냅샷 | **렌더된 UI 구조 전체** — 레이아웃·메뉴·섹션·컴포넌트 |
| 이미지 | 원본 그대로 내려받기 | **브랜드 컨셉에 맞춰 새로 생성해 교체** |
| 브랜드 | 손대지 않음 | **브랜드명·로고·톤 전면 교체** |
| 편집 | 없음 | **WYSIWYG(위지윅) 블록 에디터로 재디자인** |
| 출력 | Blob에 저장 | **새 웹사이트로 발행(배포)** |

핵심 한 줄: "임의의 사이트를 편집 가능한 템플릿으로 뜯어와서, 내 브랜드로 갈아입히고, 위지윅으로 고쳐 발행한다."

---

## 2. 전체 파이프라인 (4단계)

```
[A] Capture        [B] Rebrand         [C] Edit           [D] Publish
구조 클론     →    브랜드/이미지 교체 →  WYSIWYG 편집   →   발행/배포
─────────         ─────────────       ──────────         ─────────
렌더링·DOM 직렬화   브랜드명 탐지·치환    블록 재배치        정적 번들 생성
CSS 인라인화       이미지 슬롯 재생성    인라인 텍스트 편집  서브도메인 발행
에셋 재호스팅      팔레트/폰트 리매핑    이미지 재생성      (선택)커스텀 도메인
섹션 트리 분해                          반응형 미리보기
```

각 단계 산출물은 하나의 **프로젝트 문서(project document)**로 누적 저장되어, 다음 단계의 입력이 된다.

## 3. Phase A — 구조 클론 (Capture)

### 3.1 왜 현재 방식으로는 안 되는가
v1의 정규식 추출은 (1) JS로 렌더되는 SPA를 못 읽고, (2) 실제 레이아웃(그리드/플렉스 계산 결과)을 모르며, (3) 외부 CSS·폰트를 따라가지 않는다. **헤드리스 브라우저 렌더링이 필수.**

### 3.2 렌더링 엔진
- **Playwright + Chromium** 으로 대상 URL을 실제 렌더 → `networkidle` 대기 → 완성된 DOM 확보.
- 호스팅 방식 3안:
  - **(권장) 별도 렌더 워커** (Render/Fly/Railway 상시 서버) + 큐. Vercel 서버리스는 실행시간·바이너리 제약이 커서 무거운 렌더에 부적합.
  - 서버리스 내 `@sparticuz/chromium` + `playwright-core` — 가능하나 콜드스타트·60초 제한 리스크.
  - 외부 렌더 API(Browserless 등) — 가장 빠른 착수, 비용 발생.

### 3.3 캡처 산출물
1. **정적 HTML**: 렌더 완료된 DOM을 직렬화. 스크립트·트래커·광고 제거(보안·경량화).
2. **CSS**: 외부 스타일시트 수집 → 인라인/번들화. (선택) 미사용 규칙 제거(PurgeCSS류).
3. **에셋**: 이미지·폰트·아이콘·배경 → **Vercel Blob에 재호스팅**하고 문서 내 URL을 전부 재작성(원본 도메인 의존 제거).
4. **섹션 트리(structure.json)**: DOM을 의미 단위로 분해 →
   `header / nav / hero / section[] / cta / footer` 로 라벨링. 이 트리가 Phase C 에디터의 편집 블록이 된다.
5. **사이트맵(navmap.json)**: `<nav>`, `role=navigation`, 드롭다운, 다국어 스위처 등에서 **메뉴 계층 구조** 추출.

### 3.4 섹션 분해 규칙(초안)
- 최상위 자식 중 `header/footer/nav/main/section` 시맨틱 태그 우선.
- 시맨틱이 없으면 시각적 휴리스틱: 전체폭 배경 블록, 반복 카드 그리드, 히어로(첫 화면 큰 영역) 감지.
- 각 블록에 `role`, `bbox(위치·크기)`, `내부 텍스트`, `이미지 슬롯 목록`, `배경/색` 메타 부여.

## 4. Phase B — 리브랜딩 (Rebrand)

### 4.1 브랜드명 탐지
후보 신호를 종합해 원본 브랜드명 추정:
`<title>` · `og:site_name` · 로고 이미지 `alt` · 헤더 최상단 텍스트 · 반복 등장 고유명사(n-gram 빈도) · 도메인명.
→ 사용자에게 "감지된 브랜드: OOO — 맞나요?" 확인 UI 제공(오탐 방지).

### 4.2 사용자 입력
- **새 브랜드명** (필수)
- **브랜드 컨셉**: 키워드(예: "따뜻한 동네 책방"), 톤(미니멀/레트로/럭셔리…), 주요 색상(선택).

### 4.3 텍스트 치환
- DOM **텍스트 노드 전역**에서 원본 브랜드명 → 새 브랜드명 치환.
- 한국어 조사·대소문자·복합어(브랜드+접미사) 처리 규칙 적용.
- 이미지 `alt`·메타태그·`<title>`도 함께 갱신.

### 4.4 이미지 교체 (핵심)
1. Phase A에서 뽑은 **이미지 슬롯**마다 역할·종횡비·주요색·주변 텍스트를 추출.
2. 슬롯 역할 + 브랜드 컨셉 → **이미지 생성 프롬프트** 자동 작성
   (예: 히어로 슬롯, 16:9, "따뜻한 동네 책방, 목재 톤, 아침 햇살").
3. 이미지 생성 API로 대체 이미지 생성 → Blob 저장 → 슬롯 교체.
4. **로고**: 텍스트 기반 워드마크 생성 또는 심플 SVG 로고 생성.
5. 사용자는 슬롯별로 "다시 생성" / "원본 유지" / "직접 업로드" 선택 가능.

### 4.5 색·폰트 토큰 리매핑(선택)
원본 CSS의 색·폰트를 토큰화해 브랜드 팔레트로 일괄 치환 → 이미지뿐 아니라 전체 톤 통일.

## 5. Phase C — WYSIWYG 편집 (Edit)

### 5.1 에디터 선택
- **(권장) GrapesJS** — 오픈소스 위지윅. HTML/CSS를 그대로 불러오고 내보낼 수 있어 Phase A 산출물과 궁합이 좋음. 블록 드래그·인라인 편집·반응형·코드 뷰 기본 제공 → **개발 기간 대폭 단축.**
- 대안: React + `contenteditable` + 자체 블록 모델. 자유도 높지만 공수 큼(스크롤바 있는 캔버스, 실행취소, 반응형 전부 자체 구현).

### 5.2 편집 기능
- 섹션 트리를 편집 블록으로 로드 → **드래그로 재배치**.
- 텍스트 **인라인 편집**, 폰트·색·크기 조절.
- 이미지 클릭 → **"AI 재생성"** 버튼(Phase B 재호출) / 업로드 / 교체.
- 섹션 **추가·복제·삭제**, 미리 만든 블록 템플릿 삽입.
- **반응형 미리보기**(모바일/태블릿/데스크톱).
- 자동 저장: 편집 상태를 프로젝트 문서(JSON + 렌더 HTML)로 저장.

### 5.3 초보자 배려 (v1 UX 계승)
단계별 가이드·실시간 진행 표시·한국어 안내를 v2에도 유지. 위지윅은 "블록을 끌어다 놓기"처럼 직관적으로.

---

## 6. Phase D — 발행 (Publish)

1. 편집 문서 → **정적 HTML/CSS/에셋 번들** 생성(자기완결형, 외부 의존 없음).
2. 발행 대상 2안:
   - **Blob 정적 호스팅 + 서브도메인** (`{slug}.myapp.com`) — 즉시·저비용.
   - **Vercel 프로젝트 자동 생성/배포** — 커스텀 도메인·재배포 파이프라인 활용.
3. (선택) 커스텀 도메인 연결, 발행 버전 이력 관리(롤백).

## 7. 데이터 모델 (Neon Postgres, 초안)

```
projects            프로젝트(1 사이트 클론 단위)
  id, owner, source_url, brand_original, brand_new, concept, status, created_at

  id, project_id, role, slot_key, blob_url, is_generated, aspect_ratio, prompt

  id, project_id, version, doc_json, rendered_html_blob, updated_at

  id, project_id, edit_version, target(blob|vercel), url, published_at
```

파일(HTML/CSS/이미지)은 Blob, 메타·관계는 Neon. (현 v1 인프라 그대로 확장)

## 9. 단계별 로드맵 (MVP → 확장)

- **P0 — 구조 클론 MVP**: 렌더 HTML + CSS 인라인 + 에셋 재호스팅 + 섹션 트리. (뷰어만)
- **P1 — 브랜드명 치환**: 감지·확인·전역 텍스트 교체.
- **P2 — 이미지 슬롯 AI 교체**: 슬롯 감지 → 프롬프트 → 생성 → 교체.
- **P3 — WYSIWYG 편집**: GrapesJS 통합, 섹션 트리 로드/저장.
- **P4 — 발행**: 정적 번들 → Blob 서브도메인 발행.
- **P5 — 확장**: Vercel 배포, 커스텀 도메인, 버전/롤백, 팔레트 리매핑.

각 단계는 독립 배포 가능하도록 설계(현 v1 위에 점진 확장).

## 10. 리스크 & 가드레일 (중요)

**법적 리스크가 이 제품의 최대 이슈다.** 타사 사이트의 UI·레이아웃·콘텐츠를 복제해 재발행하는 행위는 저작권·상표권·트레이드드레스(디자인 고유성) 침해 소지가 있다. 제품을 안전하게 만들기 위한 필수 가드레일:

- **대상 제한**: 본인 소유이거나 명시적 이용 허락을 받은 사이트, 또는 오픈 라이선스 템플릿만 클론하도록 UI·약관에서 강제.
- **"출발점 템플릿" 원칙**: 결과물은 그대로 재발행하는 것이 아니라 *실질적으로 재디자인*하도록 유도(브랜드명·로고 교체 필수, 이미지 생성 교체 기본값).
- **원본 보호 자산 차단**: 원본 로고·상표·저작권 이미지/문구의 **그대로 재발행 방지** — 브랜드 자산은 반드시 교체 단계를 거쳐야 발행 가능.
- **robots.txt / 이용약관 존중**, 로그인 벽·유료 콘텐츠 스크래핑 금지.
- 발행물에 원 출처를 오인시키는 요소(원 브랜드명 잔존 등) 없도록 검증 후 발행.

**기술 리스크**: SPA 렌더 편차, 봇 차단, CSS 격리(원본 스타일 충돌), 대용량 페이지, 폰트 라이선스. → 렌더 실패·부분 실패에 대한 친절한 폴백 필수.
